# Архитектура

Приложение построено на основе разделения ответственности:

## UI Layer

Отвечает только за:

* отображение интерфейса;
* ввод параметров;
* отображение состояния обработки.

## Controller Layer

`MainWindowController`

* валидирует пользовательский ввод;
* преобразует UI данные в параметры системы;
* управляет взаимодействием UI и Orchestrator.

## Service Layer

### Orchestrator

Координатор приложения:

* запускает поиск файлов;
* управляет пайплайном обработки;
* переключает состояния приложения;
* управляет pause/resume/cancel;
* управляет таймером автоперезапуска.

### FileSearchService

* выполняет поиск файлов;
* формирует список `FileTask`.

### FileProcessingService

* выполняет XOR-обработку файлов;
* работает чанками;
* поддерживает pause/cancel.

### FileProcessor

* отвечает за безопасную запись файлов;
* создаёт temporary files;
* выполняет atomic commit;
* делает rollback при ошибках.

## Многопоточность

Приложение использует 2 потока:

| Поток        | Назначение                     |
| ------------ | ------------------------------ |
| Main Thread  | UI + Orchestrator              |
| workerThread | File services + file utilities |


## Состояния приложения


| Состояние | Описание                        |
| --------- | ------------------------------- |
| Idle      | Ожидание                        |
| Running   | Выполняется обработка           |
| Paused    | Обработка приостановлена        |
| Cancelled | Обработка отменена              |
| OnTimer   | Ожидание следующего автозапуска |


## Безопасность записи

Результат никогда не пишется напрямую в целевой файл.

Алгоритм:

1. Создаётся temporary file;
2. Данные записываются во временный файл;
3. Выполняется atomic commit;
4. При ошибке выполняется rollback.

Это предотвращает:

* Повреждение результирующих файлов;
* Частично записанные результаты;
* Потерю данных при сбое.
